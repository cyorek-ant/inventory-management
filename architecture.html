Filter Data Flow — End to End
1

User selects a filter

FilterBar.vue contains four dropdowns (Time Period, Warehouse, Category, Order Status). Each is v-model bound to a ref in the useFilters singleton composable.

2

All views react simultaneously

Every view watches the shared filter refs. Changing a filter triggers loadData() in Dashboard, Inventory, Orders, etc. — all independently, using the same filter state.

3

api.js builds the request

Composable values are mapped to query parameters and passed to Axios.

GET /api/dashboard/summary?warehouse=Tokyo&category=Sensors&month=Q1-2025
4

FastAPI filters in-memory data

The endpoint receives params, calls apply_filters() and filter_by_month() on the in-memory lists. Quarters (Q1-2025) are expanded to their constituent months before matching.

filtered = [o for o in orders if o['warehouse'] == 'Tokyo'] filtered = [o for o in filtered if o['category'].lower() == 'sensors'] filtered = filter_by_month(filtered, 'Q1-2025')
5

Pydantic validates & serializes

FastAPI runs the filtered list through the declared response_model, ensuring shape correctness and stripping any unknown fields before sending JSON.

6

Vue re-renders

The response is stored in a local ref (e.g. items.value). Computed properties that depend on it (search filtering, sorting) update automatically, triggering a template re-render.

Key Patterns

Singleton Filter State

useFilters.js is imported as a Vue composable that returns the same reactive refs to every caller. All views and the FilterBar share identical state — one filter change propagates everywhere with no prop drilling or event bus.

Raw Refs + Computed Properties

Views store raw API responses in refs (allOrders, inventoryItems). Client-side search and sorting are implemented as computed properties that derive from those refs — keeping API calls minimal and local filtering fast.

In-Memory Data Layer

All JSON files are loaded once at server startup into Python module-level variables. No database queries — all filtering is O(n) list comprehension. Suitable for demos up to ~10K items; changes reset on restart.

Locale-Aware i18n + Currency

useI18n.js provides a t() function with en/ja translations and a fallback chain. Currency is bound to locale (en → USD, ja → JPY). Locale selection persists to localStorage. Specific fields (product names, warehouses, customers) have explicit locale mapping functions.
